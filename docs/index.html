ALIGN — Why?
Define business goal and measurable outcomes
/objective · /objectives
O-NNN Objective
"What do we want to achieve and how do we measure success?"
Business goal with measurable Key Results:
• Numeric KR: metric + baseline + target + current → progress %
• Descriptive KR: description + status (NOT_STARTED | IN_PROGRESS | ACHIEVED)
scopes — thematic categories (e.g. backend, performance) — propagate down to Ideas and Tasks.
assumptions — hypotheses that must hold (Theory of Change).
appetite — effort budget: small (days) | medium (weeks) | large (months).
derived_guidelines: [G-NNN] — guidelines created FOR this objective (outbound traceability).
knowledge_ids: [K-NNN] — linked domain knowledge.
relations — depends_on, related_to, supersedes, duplicates (between objectives).
ACTIVE ➔ ACHIEVED | ABANDONED | PAUSED
On ACHIEVED/ABANDONED → reminder: review derived guidelines
▶
G-NNN Guidelines
derives Objective creates project standards
derived_from: "O-001" — traceability
scope + weight (must | should | may)
rationale + examples[]
Cross-project: guidelines import
Status: ACTIVE | DEPRECATED
COVERAGE TRACKING — 3 levels of progress measurement:
Planning
KRs with ≥1 linked Idea
"2/3 KRs covered"
Execution
DONE tasks / total
"5/8 tasks done (62%)"
Outcome
KR current vs target
"KR-1: 61%, KR-2: 89%"
scopes + KRs propagate down

EXPLORE — What and how?
Propose a solution, explore options, assess risk
/idea · /discover · /risk
I-NNN Idea
"How do we want to achieve this?"
Solution proposal. Hierarchical (parent_id).
category: feature, improvement, experiment, migration, refactor, infrastructure, research
advances_key_results: ["O-001/KR-1"] — link to KR
scopes — inherited from Objective
knowledge_ids: [K-NNN] — domain context
Relations: depends_on, related_to, supersedes, duplicates
DRAFT ➔ EXPLORING ➔ APPROVED ➔ COMMITTED
Also possible: REJECTED (with rejection_reason, merged_into)
R-NNN Research (/discover)
"What did we learn analyzing the problem?"
Structured analysis: deep-explore, deep-risk, deep-architect.
category: architecture, domain, feasibility, risk, business, technical
linked_entity → O-NNN or I-NNN  (+ linked_idea_id)
key_findings[] — key conclusions
decision_ids: [D-001, D-002] — decisions from research
file_path → full analysis markdown
DRAFT ➔ ACTIVE ➔ SUPERSEDED | ARCHIVED
▶
D-NNN Decisions (unified)
exploration options, findings, blockers
risk severity, likelihood, mitigation
standard 12 types: architecture, security, performance, …
confidence: HIGH | MEDIUM | LOW
decided_by: claude | user | imported
OPEN decisions block tasks
OPEN ➔ ANALYZING ➔ CLOSED
Risk: OPEN → ANALYZING → MITIGATED | ACCEPTED → CLOSED
Also possible: DEFERRED, reopen (CLOSED → OPEN)
idea APPROVED → /plan {idea_id}

Loop: pick task → load context → implement → validate → complete
/next | /run | /do
Repeat for each task (next / run) • multi-agent ready

LEARN — What did we learn?
Extract insights, strengthen organizational knowledge
/compound
COMPOUND /compound — Extraction
"Analyze the completed work and extract systemic insights"
Reads: the entire project — tasks, decisions, changes, research, guideline violations.
Looks for: repeatable patterns, avoided mistakes, validated decisions, architectural insights.
Produces: Lessons (L-NNN) — cross-project learning with severity and tags.
▼
L-NNN Lessons Learned
"Knowledge that outlives this project"
Categories: pattern-discovered, mistake-avoided, decision-validated, decision-reversed, tool-insight, architecture-lesson, process-improvement, market-insight
severity: critical | important | minor
tags[] — for cross-project search (lessons read-all --tags "caching,redis")
task_id, decision_ids — link to the source of learning

Lessons can be promoted → becoming permanent organizational knowledge:
➔ lessons promote L-001 ➔ Guideline G-NNN
➔ lessons promote-knowledge L-001 ➔ Knowledge K-NNN
Severity → weight: critical → must, important → should, minor → may
K-NNN Knowledge
"Domain knowledge — versioned, linked, reusable"
category: domain-rules, api-reference, architecture, business-context, technical-context, code-patterns, integration, infrastructure
source: documentation | lesson | research | user | codebase | ai-extraction  (+ derived_from_lessons)
linked_entities — tasks, ideas, objectives, guidelines (relation: required | context | reference)
dependencies: [K-NNN] — knowledge this depends on
review_interval_days — automatic staleness detection
Versioning: every content change creates a new entry in versions[] (+ change_reason)
Impact analysis: knowledge impact K-001 — scans tracker, ideas, objectives
DRAFT ➔ ACTIVE ➔ REVIEW_NEEDED ➔ DEPRECATED ➔ ARCHIVED
⟳ FEEDBACK LOOP — Closed knowledge circuit
Lesson L-NNN promoted to Guideline G-NNN → Guideline loaded into context of new tasks (by scopes) → agent applies the rule in subsequent projects → /compound verifies whether the rule works (decision-validated).

Lesson L-NNN promoted to Knowledge K-NNN → Knowledge linked to tasks and ideas (knowledge_ids) → agent has domain context without re-discovery. → Versioned: changes create history.

Guidelines imported cross-project: guidelines import {project} --source {other}
Lessons queried cross-project: lessons read-all --severity critical
Knowledge impact analysis: knowledge impact K-001 — what depends on this knowledge?

Every project strengthens the next. Knowledge is never lost.

SLASH COMMANDS — complete command reference
Command Phase Description
/do {task} Quick Single task start-to-finish, minimum ceremony (80% use case)
/objective {title} 1 Align Define business objective with measurable KRs (north star)
/objectives [id] [action] 1 Align List/show/manage objectives, update KR progress, coverage dashboard
/idea {title} 2 Explore Add idea to staging (supports --parent, --relates-to, --advances)
/ideas [id] [action] 2 Explore List/show/manage ideas (explore, approve, reject, commit)
/discover {topic|idea_id} 2 Explore Explore options, assess risks, design architecture → D-NNN + R-NNN
/risk [title|id] [action] 2 Explore Manage risks (add, analyze, mitigate, accept, close)
/research [id] [action] 2 Explore Manage research objects (R-NNN structured analysis)
/plan {goal|idea_id} 3 Plan Decompose into task DAG (draft → review → approve)
/task {description} 3 Plan Quick-add a task with alignment-driven acceptance criteria
/guideline {text} Config Add project guideline (standard, convention, rule)
/guidelines [scope] Config List/manage guidelines, import cross-project
/knowledge [id] [action] Config Manage knowledge objects (domain rules, patterns, context)
/ac-template [id] [action] Config Manage AC templates (reusable parameterized acceptance criteria)
/next 4 Execute Get and execute next task (includes verification + guidelines check)
/run [tasks] 4 Execute Execute continuously: /run, /run 3, /run T-003..T-007
/decide 4 Execute Review and resolve open decisions (unblocks tasks)
/review {task_id} 4 Execute Deep code review (optional — basic verification built into /next)
/status Info Show current project status dashboard + DAG
/log Info Show full audit trail (changes + decisions)
/compound 5 Learn Extract lessons learned from project execution
/onboard Setup Import brownfield project knowledge into Forge
/help Info Show all commands with descriptions
