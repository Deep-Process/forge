ALIGN — Dlaczego?
Zdefiniuj cel biznesowy i mierzalne wyniki
/objective · /objectives
O-NNN Objective
"Co chcemy osiągnąć i jak zmierzymy sukces?"
Cel biznesowy z mierzalnymi Key Results:
• Numeric KR: metric + baseline + target + current → progress %
• Descriptive KR: description + status (NOT_STARTED | IN_PROGRESS | ACHIEVED)
scopes — kategorie tematyczne (np. backend, performance) — propagują w dół do Ideas i Tasks.
assumptions — hipotezy które muszą się sprawdzić (Theory of Change).
appetite — budżet wysiłku: small (dni) | medium (tygodnie) | large (miesiące).
derived_guidelines: [G-NNN] — guidelines stworzone DLA tego celu (outbound traceability).
knowledge_ids: [K-NNN] — powiązana wiedza domenowa.
relations — depends_on, related_to, supersedes, duplicates (między celami).
ACTIVE ➔ ACHIEVED | ABANDONED | PAUSED
Gdy ACHIEVED/ABANDONED → ostrzeżenie: sprawdź derived guidelines
▶
G-NNN Guidelines
derives Objective tworzy standardy projektu
derived_from: "O-001" — traceability
scope + weight (must | should | may)
rationale + examples[]
Cross-project: guidelines import
Status: ACTIVE | DEPRECATED
📈 COVERAGE TRACKING — 3 poziomy mierzenia postępu:
Planning
KRs z ≥1 linked Idea
"2/3 KRs covered"
Execution
DONE tasks / total
"5/8 tasks done (62%)"
Outcome
KR current vs target
"KR-1: 61%, KR-2: 89%"
scopes + KRs propagują w dół

EXPLORE — Co i jak?
Zaproponuj rozwiązanie, zbadaj opcje, oceń ryzyko
/idea · /discover · /risk
I-NNN Idea
"Jak chcemy to osiągnąć?"
Propozycja rozwiązania. Hierarchiczna (parent_id).
category: feature, improvement, experiment, migration, refactor, infrastructure, research
advances_key_results: ["O-001/KR-1"] — link do KR
scopes — dziedziczy z Objective
knowledge_ids: [K-NNN] — kontekst domenowy
Relations: depends_on, related_to, supersedes, duplicates
DRAFT ➔ EXPLORING ➔ APPROVED ➔ COMMITTED
Możliwe: REJECTED (z rejection_reason, merged_into)
R-NNN Research (/discover)
"Co odkryliśmy analizując problem?"
Structured analysis: deep-explore, deep-risk, deep-architect.
category: architecture, domain, feasibility, risk, business, technical
linked_entity → O-NNN lub I-NNN  (+ linked_idea_id)
key_findings[] — kluczowe ustalenia
decision_ids: [D-001, D-002] — decyzje z badania
file_path → pełny markdown analizy
DRAFT ➔ ACTIVE ➔ SUPERSEDED | ARCHIVED
▶
D-NNN Decisions (unified)
exploration opcje, findings, blockers
risk severity, likelihood, mitigation
standard 12 typów: architecture, security, performance, …
confidence: HIGH | MEDIUM | LOW
decided_by: claude | user | imported
OPEN decisions blokują taski
OPEN ➔ ANALYZING ➔ CLOSED
Risk: OPEN → ANALYZING → MITIGATED | ACCEPTED → CLOSED
Możliwe: DEFERRED, reopen (CLOSED → OPEN)
idea APPROVED → /plan {idea_id}

LEARN — Czego się nauczyliśmy?
Wyciągnij wnioski, wzmocnij wiedzę organizacyjną
/compound
COMPOUND /compound — Extraction
"Przeanalizuj wykonaną pracę i wyciągnij systemowe wnioski"
Czyta: cały projekt — tasks, decisions, changes, research, guidelines violations.
Szuka: powtarzalnych wzorców, unikniętych błędów, zwalidowanych decyzji, insights architektonicznych.
Produkuje: Lessons (L-NNN) — cross-project learning z severity i tags.
▼
L-NNN Lessons Learned
"Wiedza która przetrwa poza ten projekt"
Kategorie: pattern-discovered, mistake-avoided, decision-validated, decision-reversed, tool-insight, architecture-lesson, process-improvement, market-insight
severity: critical | important | minor
tags[] — do wyszukiwania cross-project (lessons read-all --tags "caching,redis")
task_id, decision_ids — link do źródła nauki

Lessons mogą być promowane → stają się trwałą wiedzą organizacyjną:
➔ lessons promote L-001 ➔ Guideline G-NNN
➔ lessons promote-knowledge L-001 ➔ Knowledge K-NNN
Severity → weight: critical → must, important → should, minor → may
K-NNN Knowledge
"Wiedza domenowa — wersjonowana, linkowana, reusable"
category: domain-rules, api-reference, architecture, business-context, technical-context, code-patterns, integration, infrastructure
source: documentation | lesson | research | user | codebase | ai-extraction  (+ derived_from_lessons)
linked_entities — tasks, ideas, objectives, guidelines (relation: required | context | reference)
dependencies: [K-NNN] — wiedza od której zależy
review_interval_days — automatic staleness detection
Versioning: każda zmiana content tworzy nowy wpis w versions[] (+ change_reason)
Impact analysis: knowledge impact K-001 — skanuje tracker, ideas, objectives
DRAFT ➔ ACTIVE ➔ REVIEW_NEEDED ➔ DEPRECATED ➔ ARCHIVED
⟳ FEEDBACK LOOP — Zamknięty obieg wiedzy
Lesson L-NNN promowana do Guideline G-NNN → Guideline ładowana do kontekstu nowych tasków (by scopes) → agent stosuje regułę w kolejnych projektach → /compound weryfikuje czy reguła działa (decision-validated).

Lesson L-NNN promowana do Knowledge K-NNN → Knowledge linkowana do tasków i idei (knowledge_ids) → agent ma kontekst domenowy bez ponownego odkrywania. → Wersjonowana: zmiany tworzą historię.

Guidelines importowane cross-project: guidelines import {project} --source {other}
Lessons czytane cross-project: lessons read-all --severity critical
Knowledge impact analysis: knowledge impact K-001 — co zależy od tej wiedzy?

Każdy projekt WZMACNIA następny. Wiedza nigdy nie jest tracona.

SLASH COMMANDS — pełna lista komend
Command Phase Description
/do {task} Quick Single task start-to-finish, minimum ceremony (80% use case)
/objective {title} 1 Align Define business objective with measurable KRs (north star)
/objectives [id] [action] 1 Align List/show/manage objectives, update KR progress, coverage dashboard
/idea {title} 2 Explore Add idea to staging (supports --parent, --relates-to, --advances)
/ideas [id] [action] 2 Explore List/show/manage ideas (explore, approve, reject, commit)
/discover {topic|idea_id} 2 Explore Explore options, assess risks, design architecture → D-NNN + R-NNN
/risk [title|id] [action] 2 Explore Manage risks (add, analyze, mitigate, accept, close)
/research [id] [action] 2 Explore Manage research objects (R-NNN structured analysis)
/plan {goal|idea_id} 3 Plan Decompose into task DAG (draft → review → approve)
/task {description} 3 Plan Quick-add a task with alignment-driven acceptance criteria
/guideline {text} Config Add project guideline (standard, convention, rule)
/guidelines [scope] Config List/manage guidelines, import cross-project
/knowledge [id] [action] Config Manage knowledge objects (domain rules, patterns, context)
/ac-template [id] [action] Config Manage AC templates (reusable parameterized acceptance criteria)
/next 4 Execute Get and execute next task (includes verification + guidelines check)
/run [tasks] 4 Execute Execute continuously: /run, /run 3, /run T-003..T-007
/decide 4 Execute Review and resolve open decisions (unblocks tasks)
/review {task_id} 4 Execute Deep code review (optional — basic verification built into /next)
/status Info Show current project status dashboard + DAG
/log Info Show full audit trail (changes + decisions)
/compound 5 Learn Extract lessons learned from project execution
/onboard Setup Import brownfield project knowledge into Forge
/help Info Show all commands with descriptions
